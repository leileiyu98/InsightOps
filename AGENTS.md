# InsightOps 项目开发说明

## 1. 项目目标

InsightOps 是一个面向企业经营分析场景的大模型应用项目。

系统允许企业内部用户使用自然语言查询业务数据。系统需要结合业务指标定义、数据库元数据和权限规则，生成只读 SQL，执行查询，分析结果，并返回可追踪、可解释的答案。

项目的主要目标是展示以下能力：

* FastAPI 后端开发
* 企业级大模型应用架构
* Text2SQL
* RAG 与业务语义检索
* Agent 工作流
* 工具调用
* SQL 安全控制
* 权限管理
* 自动评测
* 可观测性与 Trace
* Memory
* MCP 工具集成
* Docker 部署与自动化测试

这是一个求职作品项目。

正确性、安全性、可测试性、可解释性和可维护性，比快速生成大量代码更重要。

---

## 2. 当前开发原则

项目必须按照里程碑逐步开发。

每次只实现当前明确要求的任务，不得提前实现后续阶段的功能。

除非任务明确要求，否则不要主动加入：

* 多 Agent 协作
* MCP Server
* 长期 Memory
* Redis
* Celery
* 微服务
* Kubernetes
* 复杂前端
* 多模型自动路由
* 与当前任务无关的抽象层
* 与当前任务无关的重构

优先选择能够满足当前验收标准的最简单方案，同时保留以后扩展的可能性。

不要为了展示技术而增加没有实际用途的框架或依赖。

## 3. 主要技术栈

默认使用以下技术：

* Python 3.12 或更高版本
* uv 进行 Python 环境和依赖管理
* FastAPI
* Pydantic
* SQLAlchemy 2.x
* Alembic
* MySQL 8.4
* pytest
* Ruff
* mypy
* Docker Compose
* GitHub Actions

只有在现有技术无法合理解决问题时，才可以增加新的主要依赖。

增加新的重要依赖前，必须说明：

1. 为什么需要该依赖；
2. 当前技术栈为什么无法合理解决；
3. 该依赖会带来什么维护成本；
4. 是否存在更简单的替代方案。

## 4. 架构原则

项目初期采用模块化单体架构。

代码应按照职责划分模块，但不要过度拆分。

建议保持以下边界：

* API 层：处理 HTTP 请求、参数校验和响应转换；
* 应用服务层：组织业务流程；
* 领域逻辑层：保存核心业务规则；
* 数据访问层：处理数据库访问；
* 基础设施层：处理外部模型、日志、缓存和第三方服务；
* 测试层：验证单元逻辑、集成流程和安全规则。

必须遵守以下规则：

* 不要把核心业务逻辑直接写在 FastAPI 路由函数中；
* 不要在多个模块中重复编写数据库查询逻辑；
* 大模型提供商必须通过项目内部接口访问；
* 核心业务逻辑应当能够在不调用真实大模型 API 的情况下测试；
* 外部系统返回的数据必须在系统边界进行校验；
* 大模型返回内容必须经过 Pydantic 或其他确定性方式校验；
* 不要让业务代码直接依赖某一家模型提供商；
* 不要使用隐藏的全局可变状态；
* 重要架构调整需要更新项目文档。

---

## 5. 数据库规则

数据库相关代码必须遵守以下要求：

* 使用 SQLAlchemy 2.x 风格；
* 数据库结构变更必须通过 Alembic migration；
* 不要通过手工修改数据库代替 migration；
* migration 必须能够从空数据库正常执行；
* migration 应尽量支持回滚；
* 不要在代码中硬编码数据库密码；
* 数据库配置必须通过环境变量读取；
* `.env.example` 只能包含示例变量名和安全的示例值；
* 真实 `.env` 文件不得提交到 Git；
* 测试数据和种子数据必须可以重复生成；
* 种子数据不能完全随机，应保证测试和评测结果可复现。

---

## 6. SQL 安全规则

本项目生成的分析 SQL 必须是只读 SQL。

系统不得仅依靠 Prompt 保证 SQL 安全。

在执行模型生成的 SQL 前，必须进行确定性检查。

必须禁止执行：

* INSERT
* UPDATE
* DELETE
* DROP
* ALTER
* TRUNCATE
* CREATE
* REPLACE
* GRANT
* REVOKE
* CALL
* 存储过程
* 多条 SQL 语句
* 其他可能修改数据库或系统状态的操作

分析 SQL 必须满足：

* 只允许 SELECT 或安全的只读查询；
* 必须通过 SQL 语法结构检查；
* 必须检查访问的数据表；
* 必须检查访问的敏感字段；
* 必须设置最大返回行数；
* 必须设置查询超时；
* 必须使用数据库只读账号执行；
* 必须记录查询审计日志；
* 权限不足时必须拒绝执行；
* 安全校验失败时不得通过自动重试绕过限制。

安全规则必须有单元测试和集成测试。

## 7. Agent 工作流规则

Agent 应优先采用确定、可控、有边界的工作流。

推荐流程：

1. 接收用户问题；
2. 识别用户意图；
3. 判断问题是否存在歧义；
4. 检索指标定义和数据库元数据；
5. 创建查询计划；
6. 生成 SQL；
7. 校验 SQL；
8. 执行 SQL；
9. 检查查询结果；
10. 必要时进行有限次数的重新规划；
11. 生成有数据依据的答案；
12. 保存完整运行记录。

必须遵守：

* 每次 Agent 运行必须设置最大步骤数；
* SQL 重新生成次数必须有限；
* 相同错误不能无限重试；
* 每次重试必须有明确原因；
* 工具调用参数和返回值必须结构化；
* 模型不能直接绕过应用层调用数据库；
* 模型不能自行修改权限和安全规则；
* 没有足够信息时，应澄清或明确说明无法判断；
* 不得编造数据库中不存在的数据；
* 最终结论必须能够追溯到查询结果或业务知识来源。

在单 Agent 工作流没有经过评测证明存在明显限制之前，不要引入多 Agent。

---

## 8. RAG 和业务语义规则

本项目中的 RAG 主要用于检索企业业务知识，而不只是普通文档问答。

可能检索的内容包括：

* 业务指标定义；
* 指标计算口径；
* 表说明；
* 字段说明；
* 表之间的关系；
* 数据更新时间；
* 权限规则；
* 常用查询案例；
* 业务异常说明；
* 已确认的用户纠正规则。

检索内容应尽量包含：

* 唯一标识；
* 内容类型；
* 内容版本；
* 更新时间；
* 来源；
* 适用范围。

必须遵守：

* 不要将整个数据库 Schema 无限制塞进 Prompt；
* 应根据问题只检索相关上下文；
* 检索不到关键业务定义时，不要自行编造；
* 最终回答应能够展示使用了哪些业务定义；
* 指标定义发生变化后，系统行为应通过数据和配置更新，而不是只能修改 Prompt；
* RAG 效果必须通过评测验证。

---

## 9. 模型调用规则

所有模型调用必须通过统一的内部接口。

模型调用代码需要支持：

* 请求超时；
* 错误处理；
* 结构化输出；
* 模型返回格式校验；
* Token 使用量记录；
* 请求耗时记录；
* 模型名称记录；
* Prompt 版本记录；
* 测试环境中的假模型或 Mock 实现。

不得在单元测试中调用真实模型 API。

不得把真实 API Key 写入代码、测试、README 或日志。

模型返回非法 JSON、缺少字段或不符合 Schema 时，必须返回明确错误或执行有限次数修复，不得直接信任。

---

## 10. 评测规则

每一个大模型功能都必须配套评测方案。

评测不能只判断生成文本看起来是否合理。

对于 Text2SQL，应优先比较：

* SQL 是否成功执行；
* 使用的数据表是否正确；
* 使用的业务指标是否正确；
* 查询结果是否与标准结果一致；
* 是否包含必要的过滤条件；
* 是否违反权限或安全规则；
* 最终结论是否被查询结果支持。

评测数据应包含：

* 用户问题；
* 问题类型；
* 期望使用的数据表；
* 期望使用的指标；
* 标准 SQL；
* 标准查询结果；
* 实际 SQL；
* 实际查询结果；
* 工具调用轨迹；
* 模型版本；
* Prompt 版本；
* Token 使用量；
* 响应时间；
* 错误分类。

常见错误分类包括：

* intent_error
* clarification_error
* retrieval_error
* metric_definition_error
* table_selection_error
* sql_generation_error
* sql_validation_error
* sql_execution_error
* analysis_error
* permission_error
* hallucination

任何声称“效果提升”的修改，都必须提供修改前后的评测结果。

---

## 11. 可观测性规则

每次 Agent 运行必须生成唯一的 `run_id`。

运行记录应尽量保存：

* 用户问题；
* 用户角色；
* 检索到的上下文；
* 查询计划；
* 工具调用记录；
* 生成的 SQL；
* SQL 校验结果；
* SQL 执行结果；
* 模型调用耗时；
* 数据库查询耗时；
* Token 使用量；
* 模型名称；
* Prompt 版本；
* 重试次数；
* 重试原因；
* 最终状态；
* 错误分类；
* 用户反馈。

日志应使用结构化日志。

不得在日志中记录：

* API Key；
* 数据库密码；
* 用户密码；
* 未脱敏的敏感字段；
* 其他不应暴露的凭证。

任意一次失败运行，都应该能够通过 `run_id` 定位失败阶段和原因。

---

## 12. Memory 规则

只有在任务明确要求时，才实现 Memory。

Memory 不应保存所有聊天内容。

优先保存经过确认且具有长期价值的信息，例如：

* 企业业务规则；
* 指标口径修正；
* 用户明确确认的查询偏好；
* 经管理员批准的组织级知识。

Memory 必须支持：

* 保存前确认；
* 来源记录；
* 作用域；
* 状态；
* 版本；
* 查看；
* 修改；
* 删除；
* 审计。

不得允许普通用户通过 Memory 永久植入危险指令。

不得让不同用户、组织或权限范围之间的 Memory 相互泄漏。

---

## 13. MCP 规则

只有在任务明确要求，并且现有内部工具已经具备稳定接口后，才实现 MCP。

不要为了在简历中出现 MCP 而将所有函数都改成 MCP 工具。

MCP 工具必须具备：

* 清晰的输入 Schema；
* 清晰的输出 Schema；
* 权限检查；
* 错误处理；
* 超时处理；
* 审计记录；
* 独立文档；
* 测试；
* 不可用时的降级方案。

MCP 不得绕过现有的 SQL 安全和权限规则。

---

## 14. API 开发规则

FastAPI 路由应保持简洁。

路由层主要负责：

* 接收请求；
* 参数校验；
* 调用应用服务；
* 转换响应；
* 映射错误状态码。

路由层不得直接包含复杂业务流程。

API 应：

* 使用明确的请求和响应模型；
* 提供合理的 HTTP 状态码；
* 不向普通用户返回内部堆栈；
* 对可预期错误使用结构化错误响应；
* 对重要接口编写集成测试；
* 保持向后兼容，除非任务明确允许破坏性修改。

---

## 15. 代码质量规则

代码必须遵守以下要求：

* 公共函数和方法应添加类型注解；
* 使用清晰、具有业务含义的命名；
* 函数职责应单一；
* 避免过长函数；
* 避免重复代码；
* 避免无必要的抽象；
* 避免为了未来可能需求而提前设计复杂架构；
* 优先使用领域相关异常；
* 注释应解释为什么这样做，而不是重复代码本身；
* 不要静默吞掉异常；
* 不要使用宽泛的 `except Exception` 隐藏错误；
* 不要为了让测试通过而删除或降低测试要求；
* 不要在未说明原因的情况下关闭类型检查或 lint 规则。

---

## 16. 测试规则

项目测试至少分为：

* 单元测试；
* 集成测试；
* 安全测试；
* 评测测试。

测试应做到：

* 测试结果可重复；
* 不依赖真实模型 API；
* 不依赖生产数据库；
* 不依赖无法控制的外部服务；
* 对关键安全边界进行明确断言；
* 对错误路径进行测试；
* 对数据库 migration 进行验证；
* 对模型非法输出进行测试；
* 对权限不足场景进行测试。

不要只追求覆盖率数字。

关键业务流程、安全限制和权限规则必须有明确测试。

---

## 17. 开发任务执行流程

开始任何代码修改前，必须：

1. 阅读本文件；
2. 阅读相关项目文档；
3. 检查现有代码和测试；
4. 明确本次任务范围；
5. 给出实施计划；
6. 说明将修改哪些文件；
7. 说明关键设计决策；
8. 说明风险和假设；
9. 说明测试方案。

如果任务要求先规划，则在完成计划前不要修改代码。

完成代码修改后，必须：

1. 运行格式检查；
2. 运行 Ruff；
3. 运行 mypy；
4. 运行相关单元测试；
5. 运行相关集成测试；
6. 运行安全测试；
7. 检查最终 Git diff；
8. 检查是否存在无关改动；
9. 更新相关文档；
10. 汇报实际执行结果。

不得声称某个命令执行成功，除非实际运行过该命令。

不得忽略失败的测试。

如果因为环境限制无法执行某个命令，必须明确说明，而不是假装已经验证。

## 18. Git 和文件管理规则

每次任务应保持修改范围清晰。

不得提交：

* `.env`
* API Key
* 数据库密码
* 临时日志
* IDE 缓存
* Python 缓存
* 大型无关文件
* 测试生成的临时文件
* 本地数据库数据目录

修改完成后，应检查：

```bash
git status
git diff
```

不要修改与当前任务无关的文件。

不要删除用户已有代码，除非任务明确要求，并且说明删除原因。

不要重写整个模块来解决一个局部问题，除非现有结构确实无法满足需求。

* 数据库权限；

* 敏感数据；
