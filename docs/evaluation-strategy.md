# InsightOps Evaluation 策略

## 1. 文档目的与阶段边界

本文定义 InsightOps 面向 InsightCloud 六个业务域的长期评测原则、评测对象、用例结构和结果判定方式，为后续 Evaluation 数据集和自动化框架提供设计依据。

M0.8 只编写策略，不创建评测数据库、业务表、种子数据、标准 SQL、评测代码、模型调用或 Agent 实现。具体阈值必须在后续具备可运行基线和经确认业务口径后设定。

M0.9 已在 [`business-definitions-v1.md`](business-definitions-v1.md) 提供首个冻结的指标与分析口径版本。后续 Evaluation 数据集和 Gold 结果必须显式引用该定义版本，未被其覆盖的候选能力仍不得自行补充口径。

## 2. 评测目标

Evaluation 用于回答以下问题：

- 系统是否正确理解了用户的问题和业务身份；
- 系统是否发现并处理了指标、时间、主体或范围歧义；
- 检索结果是否包含正确且有效的业务定义与数据元信息；
- 查询计划是否选择了正确业务域、指标、粒度、时间和连接路径；
- 生成的只读查询是否安全、可执行并符合权限；
- 查询结果是否与标准结果一致；
- 最终结论是否完全得到查询结果和业务知识支持；
- 系统在不知道、无权限、数据不完整或定义冲突时是否正确拒绝或澄清；
- 新版本相对已发布基线是否真实改善，是否引入安全或业务回归。

评测不以语言是否流畅作为主要成功标准，也不以单次演示结果代替固定数据集上的可复现结果。

## 3. 被评测的能力层级

### 3.1 业务意图与歧义处理

验证系统能否识别业务域、问题类型、分析主体、指标、维度、时间范围和比较方式。对于“收入”“用户”“活跃”“转化”“商家”等可能存在多种口径的词，验证系统是否根据上下文选中唯一有效定义，或在无法唯一判断时提出必要澄清。

### 3.2 业务知识检索

验证检索是否返回正确的指标版本、对象说明、状态规则、时间语义、连接规则、数据新鲜度和权限信息；同时验证是否避免引入不相关业务域或已失效定义。

### 3.3 查询计划

验证计划是否显式包含：

- 需要使用的业务域；
- 指标和定义版本；
- 分析主体与事实粒度；
- 时间字段、业务时区和观察窗口；
- 必要的状态、维度和过滤条件；
- 跨域连接路径和防重复聚合规则；
- 权限与敏感数据要求；
- 仍需澄清或无法回答的内容。

### 3.4 Text2SQL

验证查询是否语法有效、只读、安全、使用正确数据范围，并与标准查询产生等价结果。SQL 字符串完全一致可以作为辅助信号，但不作为唯一正确性标准。

### 3.5 执行结果分析

验证系统是否正确读取空结果、单值、分组、趋势、排名、漏斗和 cohort 结果，是否正确处理单位、币种、百分比、分母和时间比较。

### 3.6 最终回答

验证结论是否被查询结果支持，是否说明关键口径、时间范围、单位、数据新鲜度和限制，是否引用使用的业务定义，并避免将相关性描述为因果关系。

### 3.7 安全、权限与运行控制

验证系统是否拒绝写操作、多语句、未授权表或敏感字段访问，是否执行最大行数、查询超时和只读账号等后续安全规则，以及失败后不会通过重试绕过限制。

## 4. 评测用例范围

### 4.1 六个单业务域

每个业务域都应覆盖指标查询、趋势、分组、状态边界、时间语义、定义检索、空结果和异常数据。

#### SaaS 产品订阅收入

重点覆盖订阅与支付状态、月付与年付、续订、升级、降级、取消、退款、客户流失与收入流失，以及账单金额、实收和收入之间的区别。

#### 商家交易市场

重点覆盖订单与订单项目粒度、创建与支付与完成状态、全额和部分退款、成交总额与净成交额与平台收入、商家状态及复购。

#### 用户增长

重点覆盖不同用户身份、注册与激活、SaaS 与市场的不同转化目标、cohort、留存窗口、完整观察期、回流和去重。

#### 营销投放

重点覆盖预算与实际花费、渠道层级、内部转化与平台报告、归因模型、归因窗口、获客成本、投入回报和无法归因流量。

#### 产品使用行为

重点覆盖事件版本、去重、活跃事件集合、功能采用、会话、关键路径、发生时间与接收时间，以及测试或机器人流量。

#### 客服支持

重点覆盖工单状态历史、首次响应与有效人工响应、解决与关闭、重开、工作日历、暂停计时、服务等级和满意度样本偏差。

### 4.2 跨业务域

跨域用例至少应覆盖：

- 营销触达到注册、激活、订阅或交易转化；
- 用户 cohort 的产品使用与后续留存；
- 产品功能采用与续订、升级、降级或取消的关联；
- 客服体验与订阅或交易结果的关联；
- 商家、交易、工单与营销之间的多粒度连接；
- 企业客户与商家存在受治理映射时的综合分析；
- 不可靠身份映射或不完整时间窗口下的拒答或降级回答。

跨域用例必须专门检查连接基数和重复聚合，不能只检查查询是否能够执行。

### 4.3 问题形态

评测集应包含：

- 单指标与多指标查询；
- 同比、环比和自定义周期比较；
- 维度分组、排序、Top N 和贡献分析；
- 漏斗、cohort、留存和复购；
- 定义、来源、版本和更新时间查询；
- 解释已知异常，但不要求无依据推断原因；
- 需要澄清的问题；
- 数据不存在或未更新完成的问题；
- 超出业务范围的问题；
- 无权限或包含敏感信息的问题；
- 诱导写入、绕过权限或暴露系统信息的对抗性问题。

## 5. 评测用例结构

每条评测用例后续至少应包含：

- 稳定的用例标识和版本；
- 用户问题与必要的对话上下文；
- 用户角色、组织范围和权限条件；
- 问题类型及所属业务域；
- 数据快照或固定数据集版本；
- 期望使用的业务对象和数据范围；
- 期望指标及指标定义版本；
- 期望维度、过滤条件、时间字段和时间范围；
- 期望连接路径及粒度约束；
- 是否需要澄清、拒绝或正常查询；
- 标准 SQL 或可独立计算的标准结果；
- 标准查询结果及比较容差；
- 最终回答必须包含和不得包含的事实；
- 期望工具调用阶段和关键参数；
- 适用的安全与权限断言；
- 主要错误类别和严重级别；
- 数据、Prompt、模型和评测规则版本；
- 人工审核说明及批准人。

实际运行记录应补充实际 SQL、实际结果、检索上下文、工具调用轨迹、模型名称、Prompt 版本、Token 使用量、各阶段耗时、重试次数、重试原因、最终状态和 `run_id`。

## 6. 数据集设计

### 6.1 可复现数据

后续评测数据应通过固定种子或受版本控制的快照重复生成，不依赖生产数据库、真实个人数据或实时外部服务。涉及金额、时间、状态和身份映射的边界值应有意设计，而不是完全随机产生。

### 6.2 数据覆盖

数据集需要包含：

- 正常业务路径；
- 取消、失败、退款、重开和迟到等反向或异常事件；
- 零值、空值、无匹配结果和仅有部分观察期；
- 不同计划、渠道、商家、功能和问题分类；
- 一对多、多对多及历史有效期关系；
- 会导致重复聚合的订单项目、归因和身份映射；
- 权限允许、部分允许和完全拒绝的范围；
- 已生效、已失效和相互冲突的知识版本。

### 6.3 数据集分层

建议后续将用例划分为：

- **开发集**：用于本地开发和可见错误诊断；
- **回归集**：固定且版本化，用于每次变更验证已知能力；
- **保留集**：限制日常暴露，用于判断是否过拟合开发用例；
- **安全集**：专门覆盖权限、敏感数据、写操作和绕过尝试；
- **挑战集**：覆盖跨域、多轮澄清、粒度陷阱和定义冲突。

同一业务场景的近似改写应放在同一数据集分区，避免问题泄漏造成虚高结果。

## 7. 评测指标

### 7.1 确定性正确性指标

- **执行成功率**：期望正常查询的用例是否成功执行；
- **结果一致率**：实际结果与标准结果在类型、行集合、数值和允许容差内是否一致；
- **业务域选择准确率**：是否选择全部必要且仅必要的业务域；
- **指标选择准确率**：是否使用正确指标与版本；
- **数据范围准确率**：主体、状态、时间、维度和过滤条件是否正确；
- **连接正确率**：连接路径、关系基数和聚合粒度是否正确；
- **澄清准确率**：应澄清时是否澄清，不应澄清时是否避免无效追问；
- **拒绝准确率**：权限、数据或定义不足时是否正确拒绝，合法问题是否未被误拒；
- **安全通过率**：危险查询被拒绝且安全查询不会因安全规则缺陷被放行；
- **答案依据率**：最终事实是否都能映射到查询结果或引用的业务知识。

### 7.2 检索指标

在存在标注相关知识条目的评测集上，衡量必要定义的召回、返回内容的精确性、有效版本命中、权限过滤和不相关上下文比例。检索文本相似但业务版本错误应判为失败。

### 7.3 运行质量指标

- 端到端响应时间及各阶段耗时；
- 数据库查询耗时；
- Token 使用量；
- 工具调用次数；
- SQL 重新生成次数；
- 每类重试原因；
- 超时率和非预期错误率。

成本和延迟只在正确性与安全门槛满足后比较，不能用更快但错误的结果声称优化成功。

### 7.4 回答质量指标

对于不能完全通过确定性规则判断的表达质量，可使用有明确 rubric 的人工审核或模型辅助审核，但后者必须经过人工校准。评价维度可包括清晰度、口径披露、限制说明、引用完整性和因果措辞。模型裁判不得替代 SQL 结果、安全和权限的确定性检查。

## 8. 结果比较规则

- 标量结果比较类型、单位、币种和允许数值容差；
- 无顺序语义的结果集按稳定键规范化后比较，不因行顺序不同误判；
- 排名和 Top N 用例必须检查排序方向、并列规则和截断规则；
- 时间序列必须检查缺失周期、时区和周期边界；
- 比率必须检查分子、分母、零分母和百分比表示；
- cohort 必须检查 cohort 起点、观察窗口和未成熟 cohort；
- 金额必须检查退款、税费、折扣、换算和小数精度；
- 最终回答中的数值、趋势和排名必须逐项对应标准结果；
- SQL 不同但结果与业务语义等价时可以判定正确，但仍需独立通过安全、权限和范围检查。

## 9. 错误分类

每次失败应尽量归因到最早出现的主要阶段，同时可以记录次要错误：

- `intent_error`：错误识别业务域、主体或问题类型；
- `clarification_error`：遗漏必要澄清或提出无必要澄清；
- `retrieval_error`：缺失必要知识、命中错误版本或引入无关知识；
- `metric_definition_error`：采用错误或未批准的指标口径；
- `table_selection_error`：选择错误的数据范围；
- `sql_generation_error`：生成不可执行或语义错误的查询；
- `sql_validation_error`：安全查询被误拒或危险查询未被阻止；
- `sql_execution_error`：数据库执行阶段失败；
- `analysis_error`：查询结果正确但分析、计算或比较错误；
- `permission_error`：错误授权、错误拒绝或敏感信息泄露；
- `hallucination`：结论缺少查询结果或业务知识支持。

必要时可附加数据质量、知识冲突、超时和基础设施标签，但不应使用宽泛的“模型错误”掩盖具体阶段。

## 10. 安全与权限评测

安全评测必须是发布门槛，而不是总分中可被其他优点抵消的一项。后续至少验证：

- `INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER`、`TRUNCATE`、`CREATE`、`REPLACE`、`GRANT`、`REVOKE`、`CALL`、存储过程及其他状态修改被拒绝；
- 多条语句和通过注释、大小写、编码或嵌套方式绕过校验的尝试被拒绝；
- 未获准业务对象、敏感字段和其他组织数据不可访问；
- 查询最大返回行数、超时和只读执行身份不可由模型取消；
- 安全失败不会触发自动改写以绕过规则；
- Prompt 注入不能修改权限、指标定义或安全规则；
- 日志和最终回答不包含密钥、密码、未脱敏敏感数据或内部堆栈；
- 合法的聚合分析不会因为过度限制产生不可接受的误拒绝。

## 11. 版本、基线与回归

每次可比较的评测运行必须固定并记录：

- 代码版本；
- 数据集和数据快照版本；
- 业务知识和指标定义版本；
- Schema 与 migration 版本；
- Prompt 版本；
- 模型名称与关键推理配置；
- 安全和权限策略版本；
- 评测器与比较规则版本；
- 运行环境和时间。

任何“效果提升”都必须在同一可比配置上提供修改前后结果、样本数量、分域结果、错误分类变化、延迟和 Token 变化。不得只报告总体均值而隐藏某个业务域或安全场景的回归。

## 12. 发布门槛原则

后续里程碑设置具体门槛时应遵守：

- 安全、权限和敏感信息泄露采用零容忍阻断项；
- 数据结果正确性和必要澄清属于核心门槛；
- 六个业务域分别达到门槛，不能由优势域抵消薄弱域；
- 跨域连接用例单独设置门槛；
- 已发布回归集不得出现未说明的严重退化；
- 延迟和成本目标不得降低安全和正确性要求；
- 未达到门槛时记录原因和限制，不通过降低标准让构建变绿。

具体数值阈值应在后续拥有首个可运行基线后由业务和工程共同确认，当前不虚构百分比目标。

## 13. 评测运行与审查流程

后续一次完整评测应依次：

1. 校验数据、知识、权限和评测配置版本；
2. 在隔离、可重复的环境执行固定用例；
3. 保存每次运行的 `run_id` 和完整阶段轨迹；
4. 运行确定性 SQL 安全、权限和结果比较；
5. 对需要语义判断的答案使用明确 rubric 审核；
6. 按业务域、问题类型、难度和错误分类聚合结果；
7. 与最近批准基线比较；
8. 人工复核所有安全失败和高严重度回归；
9. 发布包含配置、样本量、失败案例和限制的评测报告；
10. 将确认的问题转化为回归用例，但避免泄漏保留集。

## 14. 六域示例问题方向

以下只用于说明未来用例覆盖方向，不给出答案、标准 SQL 或未确认指标口径：

- SaaS 产品订阅收入：某计划在指定周期的新增与流失经常性收入如何变化；
- 商家交易市场：某月支付订单、完成订单、退款和净成交额分别是多少；
- 用户增长：某注册 cohort 在完整观察窗口内的激活和留存情况如何；
- 营销投放：某渠道在指定归因模型下带来的有效转化和成本如何；
- 产品使用行为：某功能版本发布后合格用户的采用与持续使用情况如何；
- 客服支持：某问题分类在指定工作日历下的首次响应和服务等级表现如何；
- 跨域：功能采用与后续续订是否存在可观察关联，数据是否足以支持该表述；
- 澄清：“本月收入增长多少”需要先确认订阅收入、平台收入或两者的经批准组合口径；
- 拒绝：请求查看无权限用户的客服原文或要求系统修改业务数据。

## 15. M0.8 交付边界与后续输入

本策略只确定评测应该验证什么以及如何记录。正式评测前仍需要后续里程碑提供：

- 经批准并版本化的指标定义；
- 明确的数据模型、权限模型和安全策略；
- 可重复生成的非生产测试数据；
- 标准 SQL 与独立核验的标准结果；
- 可运行的检索、Text2SQL、Agent 和评测框架；
- 首个基线结果及据此确认的量化门槛。

在这些输入具备前，不声称任何大模型功能已经通过 Evaluation。
